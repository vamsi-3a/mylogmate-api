# Alembic Migration Patterns

## Creating a Migration
```bash
# After modifying any model in app/db/models/
alembic revision --autogenerate -m "add tags table"
```

## Applying Migrations
```bash
alembic upgrade head      # Apply all pending
alembic downgrade -1      # Rollback one step
alembic current           # Show current revision
alembic history           # Show migration history
```

## Rules
- EVERY model change needs a migration. No exceptions.
- Always review autogenerated migration before applying — Alembic sometimes misses things
- Never edit a migration that's already been applied to production
- Migration messages should be descriptive: "add_tags_table", "add_email_to_users", "create_chat_sessions"
- Test migrations on a fresh DB: drop all → `alembic upgrade head`
- Include both upgrade() and downgrade() functions
- For data migrations (not just schema), write explicit SQL in the migration
- Import all models in alembic/env.py target_metadata so autogenerate detects everything

## Common Gotchas
- Adding a NOT NULL column to existing table? Provide a default or do it in 2 steps: add nullable → backfill → make non-null
- Renaming columns? Alembic detects it as drop+add. Use `op.alter_column()` explicitly.
- UUID columns need: `from sqlalchemy.dialects.postgresql import UUID` and `sa.Column(UUID(as_uuid=True))`
